# Versioning and Release Guide

This guide explains how versioning and releases work in the Intent Engine project.

---

## Overview

The project uses **Commitizen** for automated version management with the following features:

- **Automatic version bumping** based on conventional commits
- **Changelog generation** from commit messages
- **GitHub release creation** with release notes
- **Docker image tagging** with version tags

---

## Versioning Scheme

We follow **Semantic Versioning** (SemVer): `MAJOR.MINOR.PATCH`

- **MAJOR** (X.0.0): Breaking changes
- **MINOR** (x.Y.0): New features (backward compatible)
- **PATCH** (x.y.Z): Bug fixes and performance improvements

Examples:
- `0.1.0` → `0.1.1` (patch: bug fix)
- `0.1.1` → `0.2.0` (minor: new feature)
- `0.3.0` → `1.0.0` (major: breaking change or stable release)

---

## How Automatic Versioning Works

### Workflow Trigger

The auto-versioning workflow (`.github/workflows/auto-version.yml`) runs when:

1. **CI/CD Pipeline completes successfully** on `main` or `master` branch
2. **Manual trigger** via GitHub Actions UI (`workflow_dispatch`)

### Version Bump Logic

The workflow analyzes commits since the last tag and determines the bump type:

| Commit Type | Pattern | Version Bump |
|-------------|---------|--------------|
| **Breaking Change** | `BREAKING CHANGE` or `feat!:` | MAJOR |
| **Feature** | `feat:` or `feat(scope):` | MINOR |
| **Fix/Perf** | `fix:` or `perf:` | PATCH |
| **Other** | `docs:`, `style:`, `chore:`, etc. | No bump |

### Automatic Actions

When version bump is needed, the workflow:

1. ✅ Updates `pyproject.toml` version
2. ✅ Updates `__version__.py` version
3. ✅ Updates `CHANGELOG.md`
4. ✅ Creates git tag (e.g., `v0.3.0`)
5. ✅ Pushes changes to repository
6. ✅ Creates GitHub Release with changelog

## Manual Version Management

### Using the Bump Script

For manual control, use the bump script:

```bash
# Bump patch version (0.3.0 → 0.3.1)
python scripts/bump_version.py --patch

# Bump minor version (0.3.0 → 0.4.0)
python scripts/bump_version.py --minor

# Bump major version (0.3.0 → 1.0.0)
python scripts/bump_version.py --major

# Preview changes without applying
python scripts/bump_version.py --patch --dry-run
```

### Manual Steps (Without Script)

1. **Update version files:**
   ```bash
   # Update pyproject.toml
   sed -i 's/^version = ".*"/version = "0.4.0"/' pyproject.toml
   
   # Update __version__.py
   sed -i 's/__version__ = ".*"/__version__ = "0.4.0"/' __version__.py
   ```

2. **Commit the changes:**
   ```bash
   git add __version__.py pyproject.toml CHANGELOG.md
   git commit -m "chore: bump version to 0.4.0"
   ```

3. **Create and push tag:**
   ```bash
   git tag v0.4.0
   git push origin main
   git push --tags
   ```

## Commit Message Format

The automation relies on **Conventional Commits**:

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Types

| Type | Description | Version Impact |
|------|-------------|----------------|
| `feat` | New feature | MINOR bump |
| `fix` | Bug fix | PATCH bump |
| `perf` | Performance improvement | PATCH bump |
| `docs` | Documentation only | No bump |
| `style` | Formatting, no code change | No bump |
| `refactor` | Code restructuring | No bump |
| `test` | Adding tests | No bump |
| `chore` | Maintenance tasks | No bump |

### Examples

```bash
# Feature (triggers minor bump)
feat: add query router for federated search

# Bug fix (triggers patch bump)
fix: resolve database connection timeout

# Breaking change (triggers major bump)
feat!: redesign intent extraction API

feat(search)!: change ranking algorithm

# With body
fix(ranking): improve constraint satisfaction

Fixed issue where range constraints were not properly
parsed when using min/max syntax.

Closes #123
```

## Changelog Management

### File Locations

- **Root `CHANGELOG.md`**: Auto-generated by Commitizen (summary)
- **`docs/reference/CHANGELOG.md`**: Detailed changelog with migration notes

### Commitizen Configuration

```toml
[tool.commitizen]
name = "cz_conventional_commits"
tag_format = "v$version"
version_scheme = "semver"
version_provider = "pep621"
update_changelog_on_bump = true
major_version_zero = true
changelog_file = "CHANGELOG.md"
changelog_incremental = true
```

### Manual Changelog Update

If automatic changelog fails, manually update:

```markdown
## [0.4.0] - 2026-03-15

### Added
- New feature description

### Changed
- Modified behavior description

### Fixed
- Bug fix description

### Removed
- Deprecated feature removal
```

## Release Process

### Automated Release (Recommended)

1. **Merge PR to main/master**
2. **CI/CD Pipeline runs** (lint, test, integration)
3. **Auto-version workflow triggers** on success
4. **Version is bumped** and tag created
5. **GitHub Release created** with changelog
6. **Docker image pushed** with version tag

### Manual Release

1. **Ensure all tests pass:**
   ```bash
   pytest tests/ -v
   ```

2. **Bump version:**
   ```bash
   python scripts/bump_version.py --minor
   ```

3. **Review changes:**
   ```bash
   git diff
   ```

4. **Commit and tag:**
   ```bash
   git commit -m "chore: release v0.4.0"
   git tag v0.4.0
   git push origin main --tags
   ```

5. **Create GitHub Release:**
   - Go to: https://github.com/itxLikhith/intent-engine/releases/new
   - Tag: `v0.4.0`
   - Title: `Release v0.4.0`
   - Body: Copy from `CHANGELOG.md`

6. **Build and push Docker image:**
   ```bash
   docker build -t anony45/intent-engine-api:v0.4.0 .
   docker push anony45/intent-engine-api:v0.4.0
   docker tag anony45/intent-engine-api:v0.4.0 anony45/intent-engine-api:latest
   docker push anony45/intent-engine-api:latest
   ```

## Troubleshooting

### Issue: Version Mismatch

**Symptom:** `pyproject.toml` version differs from last tag

**Fix:**
```bash
# Get last tag
LAST_TAG=$(git describe --tags --abbrev=0 | sed 's/^v//')
echo "Last tag: $LAST_TAG"

# Sync version
sed -i "s/^version = \".*\"/version = \"$LAST_TAG\"/" pyproject.toml
sed -i "s/__version__ = \".*\"/__version__ = \"$LAST_TAG\"/" __version__.py

git add __version__.py pyproject.toml
git commit -m "chore: sync version with last tag $LAST_TAG"
```

### Issue: Tag Already Exists

**Symptom:** Workflow fails with "tag already exists"

**Fix:**
```bash
# Remove existing tag locally and remotely
git tag -d v0.3.0
git push origin :refs/tags/v0.3.0

# Or skip version bump if no new commits
```

### Issue: Changelog Not Updating

**Symptom:** `CHANGELOG.md` not updated after bump

**Fix:**
```bash
# Manually generate changelog
cz changelog --current-version 0.3.0 --new-version 0.4.0

# Or use incremental update
cz changelog --incremental
```

### Issue: Workflow Not Triggering

**Symptom:** Auto-version workflow doesn't run

**Check:**
1. CI/CD Pipeline completed successfully
2. Branch is `main` or `master`
3. No `[skip ci]` in commit message
4. GitHub Actions enabled for repository

**Manual trigger:**
- Go to: https://github.com/itxLikhith/intent-engine/actions/workflows/auto-version.yml
- Click "Run workflow"

## Best Practices

### Do's

✅ **Use conventional commits** consistently
✅ **Write descriptive commit messages**
✅ **Include scope** when applicable: `feat(api): add rate limiting`
✅ **Reference issues** in commits: `Closes #123`
✅ **Test before merging** to main
✅ **Review auto-generated changelog** before release

### Don'ts

❌ **Don't skip CI** unless absolutely necessary
❌ **Don't manually edit tags** after creation
❌ **Don't force push** after tagging
❌ **Don't merge to main** with failing tests
❌ **Don't use vague commits** like "fix stuff" or "update code"
